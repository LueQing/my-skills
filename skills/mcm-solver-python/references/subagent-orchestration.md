# Subagent 编排协议

数学建模任务默认启用 subagent。目标是提高吞吐，而不是把问题拆乱。确认前后的子任务权限不同，必须分阶段控制。

## 触发原则

- 默认启用 subagent。
- 人类确认前，仅允许 subagent 做模型准备，不允许提前定稿实现或论文。
- 以下情况应优先触发：
- 需要并行完成案例阅读、模型择优和合理性校验
- 同时要求题意分析、Python 实现、论文整理
- 题目附件多，数据阅读与代码实现可并行
- 当前题求解还需要参考案例、模板或历史解答
- 用户明确要求“并行”“分工”“subagent”
- 小任务不要拆分。

## 一级默认分工

### 0. 范例阅读 subagent

职责：
- 读取内置案例 `problem` 与 `solutions`
- 提取题型、数据组织、章节结构、图表表达和常见方法路线
- 形成可供其他 subagent 使用的案例参考

交付：
- 案例结构摘要
- 可借鉴的方法路径
- 不可直接复用的内容边界

不负责：
- 替当前题直接选模型
- 生成当前题正文

### 1. 模型择优 subagent

职责：
- 基于当前题关键词、约束和案例参考，比较候选模型
- 给出推荐模型和不优先理由

交付：
- 候选模型清单
- 推荐顺序与适用条件
- 需要人类确认的分歧点

不负责：
- 跳过人类确认直接拍板
- 提前进入完整实现

### 2. 合理性校验 subagent

职责：
- 检查推荐模型是否满足目标、约束、数据条件和 Python 可实现性
- 提前暴露维度、边界条件、求解代价和数据不足等风险

交付：
- 风险清单
- 前提条件清单
- 对人类确认问题的补充建议

不负责：
- 替代主 agent 做最终裁决
- 生成最终实验结论

### 3. 题意/建模 subagent

职责：
- 重述题目目标
- 判断题型
- 提出建模假设
- 梳理变量、参数、约束、评价指标
- 比较 1 到 2 个可行模型并给出推荐

交付：
- 问题拆解
- 推荐模型路线
- 变量和约束清单
- 对 Python 实现的关键提醒

不负责：
- 最终代码落地
- 最终论文定稿
- 最终结论拍板

### 4. Python 实现 subagent

职责：
- 基于已确认模型编写 `.ipynb`，必要时补 `.py`
- 组织数据处理、核心求解、可视化和结果输出
- 说明依赖库、运行方式和未验证点

交付：
- 可运行脚本或 Notebook
- 文件结构建议
- 图表与结果产物清单
- 风险、假设和待确认事项

不负责：
- 擅自改模型方向
- 代替主 agent 输出最终答案

### 5. 论文整理 subagent

职责：
- 在用户需要 LaTeX 或论文交付时，按 `cumcmthesis` 结构组织内容
- 把模型、图表、表格和结果说明整理进论文框架

交付：
- 论文章节框架
- LaTeX 文段草稿
- 图表/表格插入建议
- 需要和 Python 结果核对的清单

不负责：
- 凭空扩写结论
- 脱离计算结果重建模型

## 主 Agent 协议

- 主 agent 先定总方案，再派发并行子任务。
- 人类确认前，主 agent 应优先派发“范例阅读”“模型择优”“合理性校验”来准备确认材料。
- 主 agent 必须给每个 subagent 明确：
- 任务目标
- 输入材料范围
- 产物边界
- 是否允许继续分工
- 写入范围或输出范围
- 主 agent 最终负责整合、冲突处理和一致性检查。

## 继续分工

- 只有当一级 subagent 的任务本身仍大到明显阻塞时，才继续分工。
- 继续分工必须满足：
- 子任务单一明确
- 边界可一句话说清
- 与同级或上级写入范围不冲突

推荐二级拆分：
- `Python 实现` -> `数据处理` + `核心求解` + `可视化/实验记录`
- `论文整理` -> `结构与摘要` + `图表表格落版`

谨慎拆分：
- `题意/建模` 通常不继续拆；只有题目天然包含多个独立子问题时才拆。

## 避免的问题

- 不要把最终整合权交给 subagent。
- 不要让多个 subagent 同时改同一逻辑块。
- 不要在人类确认前，让实现 subagent 盲写 Python 代码。
- 不要让论文 subagent 脱离 Python 实验结果单独产出结论。
- 不要无限递归拆分，导致协调成本高于收益。
