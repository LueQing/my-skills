# asker-collector

按以下流程处理多站点网页 AI 的收集与整理任务。

## 定位

这是一个分阶段工作流，不是全自动网页代理。

- 用户负责上传附件、粘贴 prompt、决定何时发送。
- 助手负责生成 prompt、读取浏览器现有回答、综合整理结果。
- 浏览器 MCP 负责读取现有页面状态，不默认承担上传、登录、验证码和全自动发送。

## 默认原则

- 默认优先稳定性，不追求脆弱的全自动网页操控。
- Prompt 生成与网页读取要分离，不把站点逻辑和任务逻辑写死在一起。
- 默认 `manual_send = true`，除非用户明确要求且页面稳定，才考虑半自动输入。
- 输出优先保存为 Markdown 文件，常用文件名为 `prompts.md` 与 `result.md`。
- 多站点综合时，优先提取共识、再标出分歧、最后给出推荐结论。

## 典型触发

### 1. 生成投喂内容

适用于这类请求：

- “给我一套发给 Claude / Grok / ChatGPT 的提示词”
- “我手动上传附件，你生成 prompt”
- “先别操控网页，先给我投喂内容”

此时应：

1. 明确任务目标、题型和交付偏好。
2. 生成首轮主 prompt。
3. 生成二轮追问 prompt。
4. 生成“附件识别失败时”的备用前缀。
5. 需要时写入 `prompts.md`。

### 2. 读取并整理结果

适用于这类请求：

- “三家都回答完了，请整合”
- “从浏览器读取 Claude/Grok/ChatGPT 的回答并整理”
- “帮我比较几家 AI 的方案差异”

此时应：

1. 先确认用户已经完成手动发送，且页面回答基本结束。
2. 读取当前浏览器标签页并识别目标站点。
3. 提取各站点的主要回答内容，过滤导航、侧边栏和无关 UI。
4. 输出结构化整理结果，默认写入 `result.md`。

## 建议输出模式

- `synthesis`
  - 综合共识、差异、风险和最终推荐路线
- `comparison`
  - 按站点逐项对比
- `raw_by_site`
  - 分站点摘录
- `paper_ready_outline`
  - 面向论文或正式文档的大纲输出

## 站点与扩展

- 默认站点：`claude`、`grok`、`chatgpt`
- 可扩展站点：`gemini`、`perplexity`、`kimi`、`doubao`
- 新增站点时，优先扩展“页面识别”和“回答提取”适配层，不要重写整个工作流。

## 输出结构建议

默认整理结果可使用以下结构：

```markdown
# 任务概览

# 多家共同结论

# 主要分歧与风险

# 综合方案

# 最终推荐路线

# 附录：分站点摘录
```

## 禁止默认承担的动作

- 自动上传文件
- 自动登录网站
- 自动处理验证码
- 在未经用户明确要求时自动代发消息
