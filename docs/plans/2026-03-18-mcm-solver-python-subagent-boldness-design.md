# mcm-solver-python subagent 激进度提升设计

## 背景

当前 `mcm-solver-python` 已经支持在思路准备阶段条件联动 `asker-collector`，并在用户确认后进入实现、运行和检验流程。但在实际使用中，subagent 的存在感仍然偏弱，默认执行倾向仍然偏保守，导致并行分工的优势没有稳定体现出来。

本次设计目标是把该 skill 的 subagent 策略改成“默认能并行就并行”，让主 agent 在绝大多数数学建模任务中主动采用并行分工，而不是把 subagent 当成偶尔启用的可选增强。

## 目标

- 把 `mcm-solver-python` 的默认 subagent 策略改为积极并行，而不是谨慎启用。
- 明确“超小任务”是少数降级情形，而不是默认处理方式。
- 为确认前、确认后两个阶段定义最低并行编制，减少执行时的自由裁量空间。
- 让最终输出体现各 subagent 的贡献摘要，使 skill 使用者能感受到并行带来的价值。
- 保持主 agent 的总控职责，避免并行失控或重复劳动。

## 非目标

- 不要求所有任务都机械拆成大量 subagent。
- 不改变 `asker-collector` 的联动边界。
- 不新增新的 reference 文件或 agent 类型，除非实施时发现必须补充。
- 不把主 agent 的最终判断权外包给任意 subagent。

## 设计概览

核心原则改成：

- 默认能并行就并行。
- 默认先设计并行分工，再派发 subagent。
- 默认让用户在最终结果中看到各 subagent 的分工与收敛结果。
- 只有任务极小、局部微调，或用户明确不要并行时，才降级。

这意味着 `mcm-solver-python` 的文档与默认 prompt 都要从“默认启用 subagent，但按需拆分”改为“默认积极使用 subagent，并显式说明最小编制”。

## 默认策略设计

### 1. 总体策略

- 主 agent 在接到任务后，默认先判断如何并行，而不是先判断要不要并行。
- 若任务属于标准建模题、含附件、多阶段交付、需要 notebook 结果、需要论文，默认进入并行模式。
- 若任务只是单一函数、单段说明、单文件修订、局部 LaTeX 改字句，或用户明确不要并行，才允许降级。

### 2. 降级条件

降级为低并行或不并行的情形仅限：

- 单一 Python 函数
- 单段模型说明
- 单文件局部修订
- 局部 LaTeX 修改
- 用户明确要求不要并行或不要 subagent

除上述情形外，默认采用并行。

## 阶段化最小编制

### 确认前阶段

用户确认思路前，默认最小编制为 3 个 subagent：

- `范例阅读 subagent`
- `模型择优 subagent`
- `合理性校验 subagent`

若还联动 `asker-collector`，则它作为额外的外部思路收集路径参与，不替代这 3 个确认前 subagent。

### 确认后阶段

用户确认思路后，默认最小编制至少为 2 个 subagent：

- `Python 实现 subagent`
- `Python 运行/检验 subagent`

这里的关键变化是：确认后不再默认只开一个实现 subagent，而是至少把“实现”和“运行/检验”拆开，避免代码作者自己给自己验收。

### 中大型题默认编制

若题目较大、附件较多、建模与实验链条较长，则确认后默认编制提升到 4 个方向：

- `数据处理`
- `核心求解`
- `可视化/实验记录`
- `结果检验`

主 agent 负责定义边界并做最终合并。

## 输出可见性设计

为了让使用者能感知 subagent 优势，最终输出不应只给一个统合结果，还应体现：

- 哪些 subagent 参与了工作
- 每个 subagent 的核心产出是什么
- 它们之间有哪些分歧或交接
- 主 agent 如何收敛为最终方案

这不要求展示内部推理细节，但要有足够的结构化摘要。

## 主 Agent 责任调整

主 agent 的职责强化为：

- 默认先设计并行分工
- 为每个 subagent 定义明确边界
- 在任务足够大时主动提高并行度
- 负责解决子任务冲突
- 在最终答案中显式汇总 subagent 贡献

主 agent 不应：

- 默认串行推进大任务
- 因为担心复杂而放弃并行
- 在没有边界设计时随意并行

## 文档改动范围

### `skills/mcm-solver-python/SKILL.md`

需要调整的部分：

- `默认原则`
  - 增加“默认能并行就并行”的总原则。
- `Subagent 并行策略`
  - 从保守启用改成激进默认。
  - 明确超小任务才降级。
- `Subagent 默认分工`
  - 为确认前、确认后写入最小编制。
  - 为中大型题写入扩展编制建议。
- `确认前默认 Subagent 调度模板`
  - 强调确认前默认并行派发 3 个 subagent，而不是按情况考虑。
- `继续分工规则`
  - 保留反机械拆分原则，但改成“默认先并行，再收敛”。
- `主 Agent 责任`
  - 增加对最终输出中 subagent 摘要的要求。
- `默认输出`
  - 补充 subagent 产出摘要的输出位置。

### `skills/mcm-solver-python/agents/openai.yaml`

需要调整的部分：

- `short_description`
  - 体现默认积极使用 subagent 并行分工。
- `default_prompt`
  - 明确除非任务极小，否则默认先设计并行分工再执行。

## 风险与控制

- 风险：并行度提升后出现机械拆分。
  - 控制：保留清晰的降级条件和边界约束。
- 风险：多个 subagent 角色重叠。
  - 控制：主 agent 必须先定职责再派发。
- 风险：用户看不到 subagent 带来的价值。
  - 控制：要求最终输出体现 subagent 产出摘要。

## 验收标准

- 文档明确写成“默认能并行就并行”。
- 确认前默认最小编制为 3 个 subagent。
- 确认后默认最小编制至少为 2 个 subagent。
- 中大型题的推荐扩展编制被写清楚。
- 只有超小任务或用户明确要求时才降级。
- 最终输出要求主 agent 汇总各 subagent 的产出。
