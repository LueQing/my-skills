# mcm-solver-python 与 asker-collector 联动设计

## 背景

当前 `mcm-solver-python` 把建模前期的“思路确定/模型确认”描述为主要依赖用户直接确认模型方向。这个流程对简单题足够，但不适合需要多方思路对比、网页 AI 交叉参考或先收集多来源意见再定路线的场景。

目标是让 `mcm-solver-python` 在前期阶段由主 agent 主导思路生成，并在合适时优先联动 `asker-collector`。用户确认思路后，再进入带 subagent 的 Python 实现、运行和检验闭环。

## 目标

- 默认推荐使用 `asker-collector` 汇集多方思路，但不强制。
- 用户明确不要联动时，允许 `mcm-solver-python` 本地完成候选思路生成与推荐。
- 不再把“思路确定”表达为用户自己完成，而是由主 agent 组织信息、输出候选路线并请求确认。
- 用户确认后，允许进入 subagent 协作的实现、运行、检验流程，而不是只停留在“模型准备”。
- 保持主 agent 对总方案、阶段切换和最终结论的控制权。

## 非目标

- 不为 `mcm-solver-python` 自动接管网页 AI 登录、上传、发送。
- 不把 `asker-collector` 绑定成唯一入口。
- 不新增新的工具依赖或自动化浏览器流程。
- 本轮不改案例资产、LaTeX 模板或 reference 文件结构，除非后续实施时发现必须补文档。

## 设计概览

前期“思路确定”阶段拆成两条路径：

1. 条件联动路径
   - 主 agent 先判断题目复杂度、信息缺口和是否适合多方收集。
   - 默认建议联动 `asker-collector`。
   - `asker-collector` 负责生成投喂 prompt、等待用户手动发给网页 AI、读取多方回答并综合。
   - 主 agent 基于综合结果整理 1 到 3 个候选思路，给出推荐路线与不优先原因，再等待用户确认。

2. 本地直通路径
   - 若用户明确不要联动，或任务简单到本地判断更直接，则主 agent 在本地完成关键词提取、候选模型比较、推荐与说明。
   - 即便本地直通，也必须停在“等待用户确认思路”这一步。

## 工作流设计

### 阶段 1：思路准备

- 读取题目、数据、约束和目标。
- 默认先读取内置案例，保留现有案例参考能力。
- 判断是否建议联动 `asker-collector`。
- 若建议联动：
  - 生成投喂包
  - 等待用户手动发送和收集网页 AI 回答
  - 综合多方结果
- 若不联动：
  - 直接在本地生成候选思路与推荐

### 阶段 2：思路确认

- 主 agent 输出：
  - 题目目标重述
  - 候选模型/思路
  - 推荐思路
  - 不优先路线及原因
  - 仍需确认的约束或风险
- 明确要求用户确认：
  - 是否接受推荐思路
  - 是否改用备选思路
  - 是否补充额外约束或偏好
- 在用户确认前，不允许进入 Python 实现、结果生成或论文定稿。

### 阶段 3：实现

- 用户确认后，允许启用 subagent。
- 典型 subagent：
  - 题意/建模细化
  - Python 实现
  - 数据处理与可视化
- 产物可以包括 `.ipynb`、必要 `.py`、依赖说明和中间实验记录。

### 阶段 4：运行

- 要求实际执行关键代码，而不是只给静态代码。
- 需保留：
  - 关键输出
  - 图表或表格
  - 运行日志摘要
  - 失败时的错误与回退修正

### 阶段 5：检验

- 单独定义检验阶段，不把它混入实现阶段。
- 检验内容至少包括：
  - 数值合理性
  - 维度/单位一致性
  - 约束满足情况
  - 边界条件
  - 随机方法复现性
  - 结果是否对齐题目目标
- 必要时可使用独立 subagent 做结果校验，避免实现与验收由同一子任务承担。

### 阶段 6：交付

- 在实现、运行、检验完成后再输出最终结果。
- 用户需要时再进入 LaTeX 论文整理。

## `asker-collector` 联动边界

`asker-collector` 仅负责以下事项：

- 生成投喂 prompt
- 等待用户手动向网页 AI 发送
- 读取网页 AI 已完成的回答
- 综合多方结论

`asker-collector` 不负责：

- 自动上传附件
- 自动登录
- 自动发送消息
- 直接替代 `mcm-solver-python` 做最终建模裁决

最终的思路推荐、用户确认与阶段切换仍由 `mcm-solver-python` 主 agent 控制。

## 文档改动范围

### `skills/mcm-solver-python/SKILL.md`

需要调整的部分：

- `description`
  - 增加与 `asker-collector` 条件联动的默认行为描述。
- `默认原则`
  - 增加“优先建议多方收集，但不强制”的原则。
- `先做什么`
  - 把“向人类确认模型”改成“主 agent 组织候选思路并请求用户确认”。
  - 增加 `asker-collector` 条件联动说明。
- `工作流`
  - 重写为“思路准备 -> 用户确认 -> 实现 -> 运行 -> 检验 -> 交付”。
- `Subagent 并行策略`
  - 将确认前 subagent 限定在思路准备层。
  - 将确认后 subagent 明确扩展到实现、运行、检验层。
- `确认前默认 Subagent 调度模板`
  - 增加 `asker-collector` 参与时的汇总位置与输出结构。
- `Python 代码要求`
  - 增加实际运行与校验要求。

### `skills/mcm-solver-python/agents/openai.yaml`

需要调整的部分：

- `short_description`
  - 体现“条件联动 `asker-collector` + 确认后进入实现/运行/检验”。
- `default_prompt`
  - 明确默认先判断是否应联动 `asker-collector`。
  - 明确确认后进入 subagent 实现、运行和检验闭环。

## 风险与控制

- 风险：联动描述过强，导致简单题也被迫走网页 AI 流程。
  - 控制：明确这是“默认建议”而非强制流程。
- 风险：`asker-collector` 与 `mcm-solver-python` 责任边界重叠。
  - 控制：把网页多方收集限定为思路准备辅助，不承担最终裁决。
- 风险：实现阶段仍停留在“写代码但不运行”。
  - 控制：单独定义运行和检验阶段，并写入默认 prompt。

## 验收标准

- `mcm-solver-python` 文档明确表达：
  - 默认推荐联动 `asker-collector`
  - 用户可明确拒绝联动
  - 思路由主 agent 生成并请求用户确认
  - 确认后进入 subagent 的实现、运行、检验流程
- `openai.yaml` 默认提示词与上述行为一致。
- 整体描述不存在“用户自己先确定思路再让 skill 执行”的旧表述。
